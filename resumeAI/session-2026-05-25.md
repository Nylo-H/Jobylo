### 1. Images sur les annonces (jobs)
- **Entité `JobOffer`** : ajout de `@ElementCollection List<String> images` avec table `job_images`
- **FileStorageService** : refactor pour accepter un sous-dossier (`store(file, subDir)`), création auto des dossiers `uploads/kyc/` et `uploads/jobs/`

- **Endpoints** :

  - `DELETE /api/jobs/{jobId}/images?imageUrl=...` → supprimer une image

### 2. Notifications temps réel (messagerie)
- **DTO `NotificationEvent`** : événement léger poussé sur `/topic/notifications/{userId}`
- **MessageServiceImpl** : injection de `SimpMessagingTemplate` ; après chaque `startConversation` et `sendMessage`, push :
  - Message complet vers `/topic/messages/{conversationId}`
  - Notification vers `/topic/notifications/{senderId}` et `/topic/notifications/{receiverId}`
- **MessageWebSocketController supprimé** : le `@MessageMapping` redondant est retiré ; tout passe par REST + SimpMessagingTemplate
- **Architecture** : point d'entrée unique (REST), logique métier centralisée dans le service, temps réel assuré par le push

### 3. Système de notation (ratings)
- **Entité `Rating`** : contrainte UNIQUE(rater_id, job_id) — un utilisateur ne peut noter qu'une fois par job
- **Enum `RatingTarget`** : `WORKER`, `CREATOR` (qui est la cible de la note)
- **DTOs** : `RatingRequest` (score 1-5, commentaire), `RatingResponse` complet
- **Repository** : `findByTarget`, `findByJob`, `existsByRaterAndJob`
- **Service** `RatingServiceImpl` :
  - Vérification KYC + job DONE + unicité + participants valides
  - Recalcul automatique de `averageRating` et `totalRatings` sur le User cible après chaque note
  - Audit log `SUBMIT_RATING`
- **Endpoints** :
  - `POST /api/ratings` → soumettre une note (KYC requis)
  - `GET /api/ratings/user/{userId}` → notes reçues par un utilisateur (public)
  - `GET /api/ratings/job/{jobId}` → notes d'un job spécifique (authentifié)
  - `GET /api/ratings/mine` → notes que j'ai reçues (authentifié)
- **User modifié** : ajout de `averageRating` et `totalRatings` (stockés, recalculés à chaque nouvelle note)
- **JobServiceImpl modifié** : `job.setWorker(null)` supprimé sur DONE — le worker reste assigné pour l'historique et la notation
- **DTOs mis à jour** : `UserResponse` et `MeResponse` incluent `averageRating` et `totalRatings`

## Problèmes résolus
- Contrainte `action_logs_action_check` violée à cause de l'ajout de `START_CONVERSATION` dans l'enum `ActionType`
- FileStorageService ne supportait qu'un seul sous-dossier (`kyc`) ; refactor pour `store(file, subDir)`
- Worker effacé après completion du job impossible pour la notation ; la suppression du `setWorker(null)` règle le problème

## Architecture décisionnelle
| Décision | Justification |
|---|---|
| REST + SimpMessagingTemplate plutôt que @MessageMapping | Point d'entrée unique, testable via curl, logique centralisée |
| Topic par utilisateur `/topic/notifications/{userId}` | Permet au client de recevoir les mises à jour sans polling |
| @ElementCollection pour les images | Simple, pas d'entité dédiée nécessaire pour un MVP |
| DbMigration @PostConstruct | Solution légère sans Flyway/Liquibase |
| averageRating stocké + recalculé | Évite un calcul coûteux à chaque affichage de profil |
| Worker gardé sur DONE | Permet l'historique + notation après complétion |
| Contrainte UNIQUE(rater, job) | Empêche la double notation sur le même job |
